Engineer Guild Hackathon 2025 で3位を受賞した話

作成日: 2025年12月28日、最終更新日: 2026年2月23日

はじめに

今更ながら、去年の9月に参加した Engineer Guild Hackathon (EGH) 2025 で3位を受賞した話 を書き残しておこうと思います。

図1. チーム12 API (アジフライ・パセリ・イカリング) 集合写真

参加したハッカソンについて

ハッカソンの概要

テーマ：「最高のエンジニアは誰だ？それを決めようじゃないか。」

Engineer Guild Hackathon (EGH) は、外資就活ドットコムが主催する、2週間で「技術選定・開発力・ユーザー体験」を競うハッカソンです。今大会は4回目の開催で、延べ46チームが参加しました。

図2. Engineer Guild Hackathon 2025 (©︎外資就活ドットコム)

キックオフは 9/6(土) にオンラインで実施され、発表された「お題」をもとに、予選発表までの一週間、プロダクトの構想からデモアプリの開発を行いました。予選発表では全チームが3グループに分かれ、各チーム7分の発表をします。審査の結果、各グループから上位5チーム、計15チームが決勝に進出し、残りの1週間でプロダクトのさらなるブラッシュアップを行います。

決勝発表は 9/20(土) にアークヒルズクラブ (東京都港区赤坂) にて対面で実施され、各チーム15分の持ち時間で「プレゼン + デモ」の発表を行いました。審査員はAtCoder、M3、日本マイクロソフトなどの名だたるIT企業から参加し、「実用性」「創造性」「ユーザビリティ」「技術的挑戦」「将来性」「完成度」「プレゼンテーション」の7項目において評価が行われました。

審査基準

各項目10点満点 (計70点満点)で、各審査員が70点ずつ持っています。

  1. 実用性：提案されたプロダクトが実際にユーザーに役立つか。
  2. 創造性：アウトプットに独創的なアイデアや新しいアプローチがどれだけ反映されているか.
  3. ユーザビリティ：ユーザーが直感的に使えるデザインと機能性を持っているか。
  4. 技術的挑戦：プロダクト開発において使用した技術が適切に選定されているか。 新しい技術や手法に挑戦しているか。
  5. 将来性：プロダクトの価値をさらに高める中長期的なグロース構想 (追加するべき機能や収益性など)をどこまで描けているか。
  6. 完成度：提案されたプロダクトの完成度や動作するプロトタイプの品質。発表時にはデモの紹介も行って欲しい。
  7. プレゼンテーション：チームのプレゼンテーションが、プロダクトの魅力や価値をしっかりと伝えられているかどうか。

EGH2025のお題

お題：「テクノロジーで、「学び」の体験を変える」

今回のお題はいわゆる「EdTech」に関するものでした。この領域は、社会的意義は大きいものの、「ICTリテラシー格差」や「顧客が価格に敏感」など、ビジネス化の難易度が高く、利用継続率・教育効果の定量化が課題となります。

「難易度が高い分、残された課題が多いからこそ、イノベーションの余地が大きい」と審査員の方はおっしゃっていました。

プロダクトの紹介

私たちが開発したのは、道端の「どうして？」をAIカメラと問いで紐解く散歩地図アプリ「holo（ホロ）」です。

主な機能

自分の「どうして？」を深める

ふと気になったものを撮影して質問を投稿すると、AIが「質問に対する回答」＋「それにまつわる問いかけ」を返します。AIからの「問いかけ」に応えることで、より深い理解を得ることができます。

また、AIの回答には根拠となった情報のURLが提示されるため、信頼性の確認と、さらなる深掘りが可能です。

図3. 自分の「どうして？」を深める

みんなの「どうして？」を知る

他のユーザーが地図上に残した「はっけん」を地図や一覧から見ることができ、自分一人では気づけなかった知識に触れることができます。また、検索絞り込み機能を使えば、近所の野良猫の行動範囲や植生の分布なども簡単に可視化できます。

図4. みんなの「どうして？」を知る

手軽な共有機能

Webアプリとして構築したため、アプリのインストールは不要です。LINEなどでリンクを送るだけで、お気に入りの「はっけん」を友人に簡単に共有できます。また、ユーザーによる投稿が「地域名 × 生物名」という質の高い検索キーワードとなり、検索結果に表示されるようになります。

図5. 手軽な共有機能

検索とおすすめ機能

ベクトル検索技術により、ユーザーの投稿をより効率的に発見できます。特に、「暗い」や「マンホール」といった、写真に移る情景・被写体の検索も可能となっています。

また、投稿の詳細画面では、その「はっけん」に関連したおすすめ投稿も確認することができます。

    図8. holo アーキテクチャ

    1. Webアプリの選定とSEO対策

    我々は、本アプリには、まず「使ってもらうハードルの低さ」が重要である考えました。そこで、アプリのインストールが不要で、URLだけで簡単に「はっけん」の共有ができるよう、Webアプリを選択しました。これにより、LINE等のSNSでの情報の拡散を促進しています。

    さらに、フロントエンドには SSR (サーバーサイドレンダリング) の観点から Next.js を採用しました。これにより、ユーザーの投稿がそのまま「地域名 × 生物名」 という質の高い検索キーワードとなり、SEOに強いコンテンツが自動生成される仕組みを構築しました。

    2. Dockerコンテナ化とGCPデプロイ

    フロントエンドとバックエンド（Flask）でサービス毎にコンテナを完全に分離し、メンテナンス性を向上させました。開発時は DevContainer を用いて開発環境を共通化しました（メンバーのPCスペックによってはDockerビルドが重くなり、トラシューのたびに私の作業が止まる…というハプニングもありましたが 笑）。ライブラリやVSCodeの拡張機能などをメンバー間で一括管理できたのは、長い目で見れば作業効率の向上に寄与できたと思っています。

    予選発表の時点では、 ngrok を使ってローカルのバックエンドをホスティングし、動くデモを完成させていた点は大きな強みでした。デプロイ時はコンテナごとに Cloud Run でサービスを展開するため、ngrokからGCP本番環境への移行も割とスムーズにできたのではないかと思ってます。

    3. データベースとバッチ処理

    データベースには Cloud SQL の PostgreSQL を使用しています。実は当初、バックエンドをGCPではなく Supabase にデプロイしようと考えていた名残です。最終的に検索機能などで GCP (Vertex AI Search) に統一することになったため、結果論ですが「使い慣れている MySQL でもよかったな」と思ってます。

    また検索インデックスの更新には、Cloud Scheduler と Cloud Run Job を用いて定期的にデータベースから JSON Lines を生成し、Vertex AI へ連携する裏側のアーキテクチャもしっかり組んでいます。

    4. プライバシー保護とセキュリティ

    位置情報を扱うSNS系アプリだからこそ、安全対策も徹底しました。

    • プライバシー保護： 他ユーザーの投稿は、作成から15分経過したものだけを地図上に表示することで、リアルタイムな位置情報などのプライバシーを保護しています。また、投稿にはユーザー名を表示しないことで身元の特定を防止しました。
    • セキュリティ： API呼び出しを登録済みトークン所持者に限定して不正利用を遮断。AIへの質問内容もフィルタを経てプロンプトインジェクション攻撃などの悪意混入を抑止し、GCP Secret Manager による機密情報の堅牢化も行っています。

    プロジェクトの振り返りと総括

    2週間という限られた期間の中で、チームとして何を優先し、どこでつまずいたのか。そしてこの開発を通して何を感じたのかを、3つのテーマに分けて振り返ります。

    💡 今回の開発で大切にしたこと

    • 徹底したリポジトリ運用と品質管理

      現時点での「自分の中の最適解」に近い運用が実現できたと思います。フロント・バック共にテストコードを整備し、リーダーが必ずプルリクエストをレビューする体制を徹底。開発終盤、多くのチームが時間に追われてブランチ運用を崩したり、main ブランチで直接コード編集したりする中で、最後まで運用方針を貫き通したことは、コード品質の面で大きな強みになったと思います。

      一方で、IssueやProjects機能は一切使えていなかったため、次回はタスク管理も GitHub 上で完結させ、よりシームレスな進捗管理を目指したいです。

    • 「情報の透明化」を支えるドキュメント管理

      docs/ 配下に環境構築マニュアル、議事録、技術調査報告書を集約し、開発の経緯や仕様を常に可視化しました。特に Dev Container を活用した 環境構築 や デプロイ手順 を詳細に言語化したことで、メンバーの環境構築コストを大幅に削減し、開発へのスムーズな合流をサポートしました。

    • フロント・バック間の連携ロスを最小化

      APIサーバーの構築と並行して、backend/ 配下に詳細なリファレンスマニュアルを整備。さらに、バックエンドの具体的な呼び出し例を examples/ として用意したことで、フロントエンド開発者が実装に迷わず集中できる環境を整えました。

    • Docker による開発環境の構築

      チーム全体のライブラリや拡張機能の差異を解消するため、Docker を用いた環境構築を徹底しました。フロントとバックでコンテナを分け、Docker Network で橋渡しをすることで、サーバー間通信の実装もスムーズに行えました。ただし、一部メンバーのPCスペックに依存するビルドの遅延や、それに伴うトラブルシューティングに時間を取られた点は、今後の環境設計における教訓となりました。

    • PoC（概念実証）の実施

      ハッカソンという短期間ながら、計36名（学生25名、社会人10名、その他1名）のユーザーに実際にアプリを届けてアンケートを実施しました。参加チームの中で唯一、実ユーザーによるテストサイクルを回せたことは大きなアドバンテージとなり、生の声に基づいた説得力のあるレビューをプレゼンに反映させることができました。

    🔥 反省点・次に活かすべき教訓

    • develop ブランチを作らなかった

      開発初期に、何も考えず「develop ブランチはいらないでしょ」と言ったことが、後に大問題になりました。結果、feat ブランチが直接 main に繋がり、後になって main ブランチが更新されるたびに本番環境へ自動デプロイされてしまう問題 が発生。テスト環境検証ができないため、main にマージした途端に本番環境で「デグレード」が発生し、冷や汗をかく場面が何度もありました…。develop ブランチは "絶対必須" です。

    • APIフレームワークの選定ミス？

      今回は「私が使い慣れているから」という理由で Flask を用いてAPIを作成しましたが、リファレンスマニュアルを綺麗に自動生成できることを考えると、FastAPI を採用した方が圧倒的に開発体験が良かったなと後悔しています。

    • ネイティブアプリ開発の知識不足

      「インストール不要な手軽さ」を重視してWebアプリ（Next.js）にしましたが、審査員の方から「なぜネイティブアプリにしないのか？」という指摘もありました。他チームの作品を見ても、今後のためにやはりネイティブアプリ（Swift, Flutter等）の知識は絶対に持っておくべきだと痛感しました。

    • PoC時に初期データを用意するべきだった

      PoC開始時、最初は地図上に他の人の「はっけん」が用意されていない状態だったため、共有体験の面白さを100%伝えきず、アンケート調査の回答も低評価でした。事前にダミーデータや初期データを仕込んでおくべきだったと反省しています。

    • 「困りごとベース」開発の重要性

      チームメンバー全体が「課題解決」よりも「自分たちが作ってて面白いか」を重視する傾向にあり、ペルソナ設定で迷走し開発がストップする局面がありました。例えば「子供向けにした方がいいのでは？」という案が出ましたが、これにはUI設計やプログラムの大幅な修正が必要となる他、私の母親からも「子育てにこのアプリは必要ない」 との意見をもらい、結局、当初のペルソナ (大学生) に戻るという紆余曲折がありました。

      今回はチームメンバーが慣れている方法に沿って開発を進めましたが、今後は「困りごとベース」の開発を意識して、よりユーザー中心の開発を心がけたいと思います。

    💭 参加した正直な感想

    まず、周りの参加者のレベルの高さに圧倒されました。プロダクトにAIの機能を搭載するのはもはや当たり前、さらにはAIエージェントにコーディングをさせることすら当然、という次元の戦いでした。

    そんなバチバチの環境の中、朝9時前に大学に集合して、翌日の夜中1時に帰る…という生活をほぼ2週間続けました。体力的にはキツかったですが、なんやかんやで「学生の時にしか体験できない青春だな」と感じ、「モノづくりって楽しいな」と心の底から思うことができました。

    今回一番の教訓は「ドキュメントをなるべく書くべし」ということです。技術調査マニュアルを残しておくと、後で見返した時に「なぜその技術選定をしたのか？」という意図を見失わずに済みます。 見返して自分が別の開発に携わる時の資産になりますし、長期間保守運用していく上では絶対に欠かせない要素だと身をもって学びました。(今、この記事を書いている最中も役立っています☺️)

    そしてなにより、私自身の当初の目的であった「北大で開発仲間を作る」ということが達成できてよかったです。これからも、新しくできた仲間と共に、どんどんハッカソンにチャレンジしていきたいです！
